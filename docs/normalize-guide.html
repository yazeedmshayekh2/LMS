Stage 3 — run_normalization

Complete guide to normalize/pipeline.py and every module it calls: loading the extract report, build_outline, build_batches, LLM prompts, validation, and writing normalized sections plus QC reports.

run_normalization build_batches validate.py LLM + schema

Pipeline position

The ingestion runner (ingestion/runner.py) runs four stages. Normalize is stage 3; readable export is stage 4 and lives in the same package.

pipeline.py

Loads extract report → build_outline → build_batches → loop: prompts, LLM, validate, write files.

toc.py

build_outline, _parse_toc_page, render_outline_context. Feeds every batch the same outline text.

batching.py

Groups pages into NormalizeBatch objects; attaches outline_context from render_outline_context.

prompts.py + schema.py + page_guides.py

Build LLM system/user messages: schema rules, fidelity by page type, raw markdown, outline.

Provider mapping, create_llm from ai.llms.factory, invoke + strip code fences.

validate.py

Frontmatter checks; optional similarity vs PyMuPDF reference from preprocess backends.

readable_pipeline.py

Reads normalized sections; reloads outline from normalize_report.json; second LLM pass optional.

build_batches — grouping pages for the LLM

Source: batching.py. Called after build_outline. Returns list[NormalizeBatch].

Each NormalizeBatch

FieldSource
batch_idp{page_start:03d}_{page_end:03d}
page_typesTypes of pages in the group
raw_markdown"\n\n".join(page.markdown)
outline_contextrender_outline_context(outline) — same for all batches

_batch_key(page)

cover, legal, preface, toc, unknown  →  (page_type, None, None)
lesson, unit_opener, inquiry, …       →  (page_type, page_number, page_number)
Batch size in practice: Lesson pages include page_number in the key, so page 10 and 11 never share a key — each lesson is usually its own batch. normalize_batch_size=2 mainly merges consecutive unknown pages. Cover, legal, preface, and toc are always one page per batch (forced flush).

Algorithm

  1. Sort pages by page_number.
  2. For cover|legal|preface|toc: flush, add single page, flush.
  3. Else: append if same _batch_key and len(current) < batch_size; otherwise flush and start new group.

Prompts, schema & page guides

System (build_system_prompt)

Normalize to schema-compliant Markdown; never invent content; preserve TOC behavior on TOC pages.

User (build_user_prompt) — sections in order

  1. MARKDOWN_SCHEMA_GUIDE (schema.py) — chunks, frontmatter, figures, footnotes
  2. guidance_for_page_types(batch.page_types) (page_guides.py)
  3. Book outline: batch.outline_context
  4. Page range and observed types
  5. Raw stage-2 markdown: batch.raw_markdown

Page guides

Always FIDELITY_RULES (no invention). Per type: COVER_GUIDE, LEGAL_GUIDE, PREFACE_GUIDE, TOC_GUIDE, or GENERAL_GUIDE for lesson/inquiry/unknown.

Required frontmatter

book_title, grade, semester, subject, unit_no, unit_title,
lesson_no, lesson_title, content_kind, page_start, page_end,
heading_path, chunk_id
Schema output rules (summary)
  • Multiple chunks per page when source has clear blocks
  • ::: figure blocks; <!-- page_break: N -->
  • content_kind=key_terms for bilingual lines: Arabic | English
  • Stable chunk_id e.g. u1-l1-body-p10

For each batch in batches:

build_outline — what it returns

Source: src/pipeline/ingestion/normalize/toc.py. Input: list[RawPageMarkdown] from the extract report. Output: BookOutline with book metadata and a flat entries: list[TocEntry].

book_titleFrom cover page 1 (or default)
grade / semesterHardcoded 8 and 1 today
entriesOne row per TOC line (printed page → title)
def build_outline(pages: list[RawPageMarkdown]) -> BookOutline:
    # 1. book_title from page 1
    # 2. entries from all pages where page_type == "toc"
    # 3. assign unit_no / lesson_no on unit & lesson kinds only
    return BookOutline(...)

Step 1 — Book title from cover

_book_title_from_cover scans layout blocks[].lines on page 1 (not the rendered markdown string).

if "العلوم" in line:  return "العلوم"
return "علوم"           # default
OCR / kashida caveat: Cover text like العـلــوم often does not contain the plain substring العلوم, so reports may show book_title: علوم even when the printed title is “العلوم”.

Step 2 — Parse TOC pages

Only pages with page_type == "toc" (from preprocess classify_page, usually pages 3–4) are parsed. _parse_toc_page walks every line in every block in order.

Line state machine

Line patternAction
^\d{1,3}$ only (e.g. 5, 10)Flush previous entry; set pending_page
Any other line while page pendingAppend to title lines for that entry
End of pageFlush last pending entry

Example from extract report (page 3)

Block lines (simplified):
  "5"     →  start entry, page_number = 5
  "المقدّمة"  →  title for page 5
  "6"     →  flush page 5; start page 6
  "): الوراثة والتكاثر1( الوحدة"  →  title for page 6
  "10"    →  flush page 6; start page 10
  "): المادة الوراثية1( الدرس"   →  title for page 10

Title text is _infer_title: non-empty stripped lines joined with a single space.

stateDiagram-v2
  [*] --> Idle
  Idle --> Pending: digit-only line
  Pending --> Pending: title line
  Pending --> Idle: flush on next digit or EOF
  note right of Pending: TocEntry(page, title, entry_kind)

Step 3 — Classify entry_kind

_infer_entry_kind(title) uses Arabic keyword regexes. First match wins (order matters).

OrderRegex / keywordentry_kind
1الوحدةunit
2الدرسlesson
3الإثراء · التوسعenrichment
4استقصاء · استكشافinquiry
5مراجعةreview
6المقدمةpreface
—no matchunknown
Ordering trap: مراجعةُ الوحدة matches الوحدة before مراجعة, so it becomes entry_kind: unit in normalize_report.json, not review. OCR-broken preface titles may stay unknown if المقدمة does not appear in the string.

Step 4 — Assign unit_no and lesson_no

A second pass walks entries in list order (TOC page order in the extract report, then line order).

entry_kindunit_nolesson_no
unitIncrement global counter; reset lesson counter—
lessonCurrent unit (or null if no unit yet)Increment within current unit
othernullnull

Numbers in parentheses in the printed TOC (e.g. الوحدة (1)) are not parsed — only positional counting in the outline list.

Step 5 — render_outline_context

Converts BookOutline to plain text embedded in every NormalizeBatch and LLM user prompt: 

book_title: علوم
grade: 8
semester: 1

TOC outline:
- page 6: unit=1 lesson=None kind=unit title=...
- page 10: unit=1 lesson=1 kind=lesson title=...

The normalization LLM uses this to fill YAML frontmatter (unit_no, lesson_title, content_kind, etc.) consistently across batches.

BookOutline.entry_for_page(n) in models.py returns the last entry with page_number <= n — useful for lookup, but build_outline itself only produces the list; the main path uses the rendered text.

build_outline — limitations

  • Depends on extract block line order — if page number and title are swapped in blocks, entries break.
  • Only processes page_type == "toc" — misclassified TOC pages are skipped entirely.
  • grade and semester are hardcoded, not read from the PDF.
  • Keyword classification is brittle under OCR (tatweel, broken المقدمة, الوحدة vs مراجعة order).
  • build_outline does not sort pages by page_number — order follows the extract report array.
